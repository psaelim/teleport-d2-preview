System states, drafted for review.

The last backlog item: loading, can’t-load, offline, and empty — one grammar for each, applied to the screens that need them. Static and self-contained; nothing in the prototype is touched. Mark up or approve, then I port.

Loading — the shape arrives first

1 · Skeleton, not spinner

Visits loading: the row geometry in motion, so nothing jumps when data lands.

Visits

The rules: skeletons mirror the exact layout they become (disc, pill, two lines = a visit row), the chrome (tabs, search) renders instantly, and no spinner exists anywhere in the app. Shimmer respects prefers-reduced-motion. Skeletons appear only after a 300ms grace period — a fast load shows nothing at all. For review: do we cache last-known data so returning users see yesterday’s truth instantly instead of skeletons? (Recommended, but it’s a platform/PWA call — DEV-FAQ open decision #1.)

2 · Busy buttons

Action pending: the label becomes the verb in progress. The button never shrinks to a spinner.

Idle → busy → done is the existing tick moment

Same size, same place; label swaps to present-progressive; disabled kills double-submit. Success is the existing full-screen moment (Requested / Booked / You’re moved) — no toast needed.

When the action fails, the button tells you where it failed

That didn’t go through. Nothing was booked — try once more.

The wrong-code grammar reused: danger text under the control, the button returns to idle, the truth (“nothing was booked”) stated outright.

One grammar for every CTA — Request the visit, Confirm booking, Move to Thu, Save. The failure line is the same .errline the wrong-code screen shipped; no new components.

3 · Can’t load

The server failed, the phone is fine. Blame lands on us, data is declared safe.

Lab results

Your results didn’t load.
That’s on us, not your phone. Nothing is lost — this screen just didn’t arrive.

If it keeps failing, we’re probably already fixing it. Check back in a few minutes.

Same shape as the search empty state (glyph, one truth, one reassurance, one action) so failure never feels like a different app. No error codes on the surface; the code goes to telemetry. For review: retry policy — manual only as drafted, or one silent auto-retry before the state shows? And do we want a status-page link for long outages?
